Phased Refactor Plan (Audit Only)

Project: docuTrust  |  Document type: Architecture audit & migration planning  |  Status: No code changes — planning only
Generated: May 14, 2026  |  Scope: E-signature PKI, sealing pipeline, verification, blockchain anchoring
Executive summary. DocuTrust implements a two-stage signing model: field capture (synchronous UI) and cryptographic sealing (async on document completion). PKI is an embedded application CA, not a standalone enterprise RCA. This document maps dependencies, risks, and a safe phased refactor toward external RCA/HSM without breaking existing documents.

6. Signing Flow Risks

  1. Deferred cryptography — UI shows signed after capture; PKI verify fails until job completes.
  2. One hash per signer — Same document hash applied to all Signature rows for a signer.
  3. Legacy completeLegacySigning — No field capture; may lack PKI signatures.
  4. Remote + local key mismatch — Capture always creates local keys; remote seal uses provider keys.
  5. Trust authorization expiry — Remote seal needs active session between capture and seal job.
  6. Silent job failure — GenerateCertificateJob catches all errors; document may show Completed without seal.
  7. Cert issued before identity final — DN from signer name/email at issuance; edits don't re-issue.
  8. DB-only revocation — No CRL; external relying parties unaffected.

9. Refactor Order (Phases 0–7)

Phase 0 — Observability (no behavior change)

  • Structured logging at seal job boundaries
  • Monitor: Completed documents with null signature_value
  • Inventory production: certificate_source mix, null transaction_id

Phase 1 — Extract boundaries (interfaces, 1:1 adapters)

  • CertificateIssuer, DocumentSealer, TrustAnchor, CertificateTrustValidator ports
  • Existing classes remain default adapters

Phase 2 — Decouple capture from issuance

  • Stop issuing certs in FieldSignatureCaptureService
  • Move getOrIssueForSigner to seal time only

Phase 3 — Split hash / seal / anchor

  • SealingService: seal only
  • DocumentHashService: hash + persist only
  • BlockchainAnchoringService: optional async job (feature flag, default off)

Phase 4 — Key store abstraction (HSM prep)

  • Key handles vs raw PEM; dual-read for legacy rows

Phase 5 — External RCA / TSP adapter

  • Map to provider_managed schema first
  • app_managed root behind legacy flag

Phase 6 — Verification & revocation hardening

  • Optional CRL/OCSP adapter; DB revocation remains UI source of truth

Phase 7 — Deprecate embedded root (long horizon)

  • Freeze getOrCreateRootAuthority for new issuances after migration window
  • Retain root rows for historical verification

11. Migration Strategy

Approach: Strangler fig + dual-write/read

  1. Classify data: app_managed_sealed, app_managed_unsealed, provider_managed, legacy_no_fields
  2. Introduce pki_backend resolution: signer → document → global (defaults unchanged)
  3. New issuances on external RCA/TSP; retain app-managed for historical verify
  4. Key migration in maintenance window: HSM for root; do not mass re-issue signer certs
  5. Blockchain: never re-hash completed docs in place; version table if re-stamping required
  6. Rollout: dev → staging snapshot → canary → cutover per signing_backend

Rollback

  • Feature flags revert to current adapters
  • PEM + certificate_authorities snapshot before HSM cutover
  • Disable external issuer → fall back to embedded CA

Compatibility rules

  • Never change signature_hash algorithm without version field
  • Never rotate root CA in place; add new CA row + superseded_by
  • Provider path first for external RCA (issuer_certificate_pem)
  • Defer CRL/SCEP/CMP until issuer adapter is stable
